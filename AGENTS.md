# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

A single-page Vue 3 + TypeScript web app (built with Vite) that helps users build format strings for [Python's `logging` module](https://docs.python.org/3/library/logging.html). Users assemble a format from log-record tokens (e.g. `%(asctime)s`) and custom text, see a live preview against sample log data, and copy the result. Deployed as static content to GitHub Pages on every push to `main` (see `.github/workflows/pages.yml`).

## Commands

- `npm run dev` — start the Vite dev server
- `npm run build` — type-check (`vue-tsc`) and build to `dist/`
- `npm run type-check` — `vue-tsc --noEmit` only
- `npm run test` — run the Vitest suite (`vitest run`)
- `npm run test:oracle` — regenerate the Python ground-truth fixtures, then run Vitest
- `npm run lint` — ESLint with `--fix` over all source files
- `npm run format` — Prettier over `src/`
- `npm run preview` — serve the built `dist/`

The `@` import alias maps to `src/` (configured in `vite.config.ts`, `vitest.config.ts` and the tsconfig files). CI (`.github/workflows/ci.yml`) regenerates the oracle fixtures on real Python 3.12, diffs them for drift, type-checks and runs the tests; `pages.yml` deploys.

## Architecture

The entire UI lives in `src/App.vue`. State is a single reactive array `selectedItems: FormattedItem[]`, a `style` (`'%' | '{' | '$'`) and an optional `datefmt`. Everything else derives from these.

**The formatting engine** lives in `src/engine/` (pure TS, no Vue, fully unit-tested) and matches real Python `logging.Formatter` semantics for all three styles:

- `engine/types.ts` — `Style`, `ValueKind`, the style-neutral `FieldSpec`, and `FormattedItem` (one element: literal text when `isText`, else a log-record field carrying a `FieldSpec`).
- `engine/parse.ts` — `parseFormat(input, style)`; one parser per style, inverse of serialize.
- `engine/serialize.ts` — `serialize(items, style)`; projects the neutral `FieldSpec` into each style's surface syntax.
- `engine/render.ts` — `renderRecord(items, record, style, datefmt?)`; renders a sample record applying real Python rules (`asctime` is computed from `created`/`msecs` in UTC via `engine/spec/strftime.ts`).
- `engine/spec/*` — the per-mechanism formatters: `printf.ts` (`%`), `formatspec.ts` (`{}` mini-language), `pyfloat.ts` (Python number formatting), `pyrepr.ts` (`str`/`repr`), `pad.ts` (width/align/grouping/zero-pad), `strftime.ts` (datefmt).

Parse and serialize are inverses, kept honest by a round-trip test. `src/parsing.ts` is a thin back-compat re-export of `engine/parse`.

**Token definitions** (`src/items.ts`): `templateItems` maps each supported log-record attribute to `{ description, kind }`, where `kind: 'str' | 'int' | 'float'` is the value's intrinsic Python type. `textItems` are quick-insert literal snippets. To add a new supported token, add it here.

**Presets** (`src/presets.ts`): named example format strings, each with a `style`, parsed through the engine when selected.

**Tests** (`src/engine/__tests__/`): `parse.test.ts` is pure (parser + round-trip); `oracle.test.ts` replays every case in `fixtures/oracle.json` — generated by `tests/oracle/oracle.py` from real Python 3.12 — through the engine and asserts parity (including that the engine raises where Python raises). Regenerate fixtures with `npm run test:oracle`.

**Sample logs** (`src/assets/logs.json`): rows of fake `LogRecord` field values used to render the live "Example Logs" preview. Keys must match `templateItems` token names.

**`ItemDialog.vue`**: a promise-based modal. `show()` returns a `Promise<EditResult>` that resolves on submit/cancel/delete; `App.vue` awaits it to add custom text or edit/delete an existing item.

**Theme** (`src/stores/config.ts`): a Pinia store persisted to localStorage (`pinia-plugin-persistedstate`) that toggles `data-theme` on `<html>`. Styling uses Pico CSS plus SCSS in `src/assets/main.scss` and per-component `<style lang="scss">`.

## The `analysis/` directory

Offline tooling, not part of the app build. `analysis/fiddle.ipynb` and `analysis/dummy-data/` (a small Python program with a `log.conf`) are used to introspect real Python `LogRecord` attributes and generate the sample data in `src/assets/logs.json` / `analysis/attributes.raw.json`. Regenerate these when updating the supported token list.
